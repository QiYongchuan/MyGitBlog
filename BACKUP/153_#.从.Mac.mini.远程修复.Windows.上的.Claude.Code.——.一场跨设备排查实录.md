# [# 从 Mac mini 远程修复 Windows 上的 Claude Code —— 一场跨设备排查实录](https://github.com/QiYongchuan/MyGitBlog/issues/153)



**背景**：Mac mini + Windows 笔记本，通过 Tailscale 组网，SSH 双向打通
**工具**：Tailscale + SSH + Claude Code
**关键词**：跨设备运维、npm 包修复、原生二进制缺失

---

## 起因

今天打开 Windows 笔记本，准备用 Claude Code 干活，结果发现——启动不了了。

诡异的是，之前没关掉的终端窗口里 Claude Code 还能正常用，但**新开的终端全部启动失败**。

这说明 Claude Code 的运行时本身没问题，是某个共享组件坏了。

而我的 Mac mini 在另一张桌子上，两台设备已经通过 Tailscale + SSH 双向打通。

那就——**坐在 Mac mini 前，远程修 Windows 吧。**

---

## 第一步：远程诊断

从 Mac mini 通过 SSH 连到 Windows：

```bash
ssh qyc@192.168.31.185
```

先看 `claude` 命令在哪、什么版本：

```bash
where claude
# 输出：C:\Users\Qyc\AppData\Roaming\npm\claude

claude --version
# 报错：此版本的 claude.exe 不兼容正在运行的 Windows 版本
```

这个报错很奇怪——之前一直用的同一个 Windows，怎么突然不兼容了？

继续查 Node.js 和 npm 的状态：

```bash
node --version    # v22.14.0 ✅
npm --version     # 10.9.2 ✅
npm list -g @anthropic-ai/claude-code
# @anthropic-ai/claude-code@2.1.114 ✅
```

Node 没问题，npm 没问题，包也装着。问题出在 `claude.exe` 本身。

---

## 第二步：发现异常

看看这个 `claude.exe` 文件有多大：

```bash
dir "C:\Users\Qyc\AppData\Roaming\npm\node_modules\@anthropic-ai\claude-code\bin\claude.exe"
# 500 字节
```

**500 字节。**

一个正常的 Claude Code 原生二进制文件应该是几百 MB 级别的。500 字节就是一个占位符。

---

## 第三步：找到根因

查看 Claude Code 的安装脚本（`install.cjs`），逻辑很清晰：

```
1. 检测操作系统和 CPU 架构（如 win32-x64）
2. 找到对应的原生二进制包（@anthropic-ai/claude-code-win32-x64）
3. 把真正的可执行文件复制覆盖掉 500 字节的占位符
```

问题出在第 2 步。检查原生二进制包：

```bash
npm list -g @anthropic-ai/claude-code-win32-x64
# @anthropic-ai/claude-code-win32-x64@0.0.0 ❌
```

**版本号是 `0.0.0`**——这是一个空壳包，里面没有真正的二进制文件。

npm 认为"装了"，但实际上是坏的。这就是为什么：
- 旧终端能用（已经加载到内存中的进程不受影响）
- 新终端不能用（需要重新加载这个损坏的 `claude.exe`）

---

## 第四步：修复

一句话解决：

```bash
npm install -g @anthropic-ai/claude-code@latest
```

2 分钟后安装完成，验证：

```bash
claude --version
# 2.1.74 (Claude Code) ✅
```

搞定。打开新终端，Claude Code 正常启动。

---

## 排查路径总结

```
启动失败
  → Windows 报"版本不兼容"（误导性报错）
    → 发现 claude.exe 只有 500 字节
      → 查看 install.cjs 了解安装机制
        → 发现原生二进制包版本为 0.0.0（空壳）
          → npm install -g 重新安装 → 修复
```

整个过程的关键在于：**Windows 报的"版本不兼容"是误导信息**，真正原因是可执行文件本身就是一个占位符，根本不是真正的程序。

---

## 几个值得记住的点

### 1. 为什么会出现 0.0.0 的空壳包？

Claude Code 的 npm 包设计是：主包（`@anthropic-ai/claude-code`）+ 平台特定的原生二进制包（如 `claude-code-win32-x64`）。原生二进制包作为 optional dependency 安装。

如果安装时网络不稳定、代理超时、或者用了 `--omit=optional`，原生包就会安装失败。npm 不会因此报错（毕竟是 optional），但会留下一个 0.0.0 的占位记录。

### 2. 为什么旧终端还能用？

已经启动的进程在内存中运行，不依赖磁盘上的可执行文件。只有新启动的进程需要重新加载 `claude.exe`，才会触发报错。

### 3. 跨设备运维的实际价值

如果我没有提前配置好 Tailscale + SSH 双向打通，修复流程会是：
1. 走到 Windows 电脑前
2. 打开浏览器搜索报错信息
3. 手动排查

而通过 SSH 远程修复：
1. 坐在 Mac mini 前不动
2. 用 Claude Code 直接 SSH 过去诊断和修复
3. 整个过程 5 分钟

**这就是"分布式大脑"的真正含义——你的算力和运维能力不绑定在一台设备上。**

## 修复命令速查

如果遇到同样的 Claude Code 启动失败：

```bash
# 1. 检查 claude.exe 大小
dir "%APPDATA%\npm\node_modules\@anthropic-ai\claude-code\bin\claude.exe"
# 如果只有几百字节，说明是占位符

# 2. 检查原生二进制包
npm list -g @anthropic-ai/claude-code-win32-x64
# 如果版本是 0.0.0，说明包坏了

# 3. 重新安装
npm install -g @anthropic-ai/claude-code@latest

# 4. 验证
claude --version
```

Mac 用户同理，只是包名换成 `claude-code-darwin-arm64` 或 `claude-code-darwin-x64`。
